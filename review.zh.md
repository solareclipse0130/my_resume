## 发现问题（按严重程度排序）

### 1. 高：公开 AI 接口缺少滥用防护（鉴权/限流）

- 位置：`src/app/api/digital-twin/route.ts:89`
- 风险：任何人都可程序化调用 `/api/digital-twin`，消耗你的付费模型额度，可能导致费用异常上涨和服务退化。
- 影响原因：该接口在服务端执行并串联你的 API Key 使用路径；若无节流和防刷，存在明显经济风险。

补救措施：

1. 增加按 IP 或会话的限流（例如每分钟 10-20 次）。
2. 在请求前增加机器人防护（Turnstile/reCAPTCHA）。
3. 增加服务端请求日志与基础异常检测（突发频率、重复负载指纹）。

---

### 2. 高：输入体积限制不足，成本与时延不可控

- 位置：`src/app/api/digital-twin/route.ts:67-87`，`:107-132`
- 风险：当前仅限制“最近 12 条消息”，但没有限制单条消息长度或总 payload 字节数；超长输入仍可能触发高成本、慢请求。
- 影响原因：异常输入（恶意或误操作）会放大 token 消耗并诱发超时。

补救措施：

1. 限制单条消息最大字符数（如 1500-2000）。
2. 限制单次请求总字符数（如 6000-8000）。
3. 对超限请求返回 `413 Payload Too Large`，并给出清晰前端提示。

---

### 3. 中：上游请求缺少超时与取消控制

- 位置：`src/app/api/digital-twin/route.ts:115`
- 风险：当 AIHubMix 变慢或无响应时，请求线程会被长时间占用，用户端表现为“卡住”。
- 影响原因：上游不稳定会直接影响生产可用性与主观体验。

补救措施：

1. 使用 `AbortController` 为 `fetch` 增加超时（建议 12-20 秒）。
2. 对超时场景返回受控错误信息。
3. 可选增加一次带抖动的重试（仅瞬时网络错误，避免重试风暴）。

---

### 4. 中：向客户端暴露了上游错误细节

- 位置：`src/app/api/digital-twin/route.ts:136-142`
- 风险：`detail: errorText.slice(...)` 可能将上游内部信息、请求标识或运维细节直接暴露给终端用户。
- 影响原因：信息泄露会提升攻击者诊断能力，也会让用户看到噪声级错误。

补救措施：

1. 客户端仅返回通用错误文案。
2. 详细错误仅写入服务端结构化日志。
3. 增加错误码分类（如 `UPSTREAM_TIMEOUT`、`UPSTREAM_4XX`、`UPSTREAM_5XX`）提升可观测性。

---

### 5. 中：每次请求都在模型提示词中携带个人联系方式

- 位置：`src/app/api/digital-twin/route.ts:24-29`（`DIGITAL_TWIN_CONTEXT` 内）
- 风险：邮箱/电话会在每次聊天请求中重复发送到第三方服务。
- 影响原因：即使信息已公开，减少重复传输 PII 仍是更稳健的隐私与合规实践。

补救措施：

1. 从系统提示词移除非必要直接标识信息（邮箱/电话）。
2. 联系方式保留在页面 UI 渲染层，不必进入模型上下文。
3. 如确需由模型输出联系方式，改为显式模板化输出，而非原始 prompt 常驻。

---

### 6. 低：聊天输入与状态提示存在可访问性缺口

- 位置：`src/components/DigitalTwinChat.tsx:168-176`，`:160-165`，`:187`
- 风险：`textarea` 缺少关联 `<label>`；加载/错误状态未通过 ARIA live region 宣告。
- 影响原因：屏幕阅读器用户的交互体验会下降。

补救措施：

1. 为 `textarea` 增加可见或 sr-only 的 `<label htmlFor=...>`。
2. 为助手状态与错误容器增加 `aria-live="polite"`。
3. 在消息视图容器加载时增加 `aria-busy`。

---

### 7. 低：快速跳转场景下客户端请求生命周期可进一步加固

- 位置：`src/components/DigitalTwinChat.tsx:69-100`
- 风险：组件卸载或路由切换时，进行中的请求未被取消。
- 影响原因：在少数情况下会出现陈旧状态更新或无效网络消耗。

补救措施：

1. 客户端每次提交使用 `AbortController`。
2. 组件卸载时取消进行中的请求。
3. 对取消态增加状态更新保护。

---

## 正向评价

1. 关注点拆分良好：UI 组件与 API 路由职责清晰。
2. API Key 保持在服务端，未暴露至浏览器代码。
3. 已有基础消息归一化与校验（`normalizeMessages`）。
4. lint/build 均通过，说明当前没有明显类型/语法回归。
5. Markdown 渲染与对话交互细节实现较完整。

---

## 建议的修复优先级

1. 先做防滥用与输入限界（问题 1、2）。
2. 再做上游超时与错误细节收敛（问题 3、4）。
3. 然后降低提示词中 PII 暴露面（问题 5）。
4. 最后补齐可访问性和请求生命周期加固（问题 6、7）。

---

本次未发现阻断编译/发布的缺陷。当前主要风险集中在运维层面（成本控制、稳定性、隐私收敛），而非立即的功能正确性问题。
